# Feature: Internationalization (pt-BR + en)

## Fase: 1 — MVP
## Prioridade: P0 (bloqueia todas as features com UI)
## Estimativa: 3.5h (infra + migração dos arquivos já escritos em pt-BR)
## Depende de: 01-project-setup

## Overview

Sistema de i18n desde o MVP. Dois locales (`pt-BR` default + `en` fallback), detecção automática via `Accept-Language`, cookie `NEXT_LOCALE` para persistir escolha, URL com prefixo por locale (`/pt-BR/...`, `/en/...`). Não é um retrofit para Fase 4 — todo texto voltado ao jogador passa por `next-intl` desde o primeiro commit que toca UI.

**Regra de ouro:** código, nomes de funções/variáveis/tipos, schema do banco, comentários, documentação interna e identifiers de key de tradução (`home.hero.title`) **sempre em inglês**. Apenas o **valor** resolvido da key varia por locale.

## Requisitos Funcionais

### Stack

- Biblioteca: `next-intl` (^3.26)
- Formato: ICU (plurais, datas, números)
- Messages: arquivos JSON em `apps/web/messages/[locale].json`
- Routing: prefixo por locale (`localePrefix: 'always'`), middleware do next-intl cuida de redirect

### Locales suportados

| Código | Nome | Uso |
|---|---|---|
| `pt-BR` | Português | Default e fallback |
| `en` | English | Navegadores em inglês, jogadores internacionais |

Fase 4 pode adicionar `es-ES`, `es-MX`. A arquitetura comporta sem retrabalho.

### Detecção de locale

Ordem de decisão no middleware (next-intl padrão):
1. Cookie `NEXT_LOCALE` se presente e válido
2. Header `Accept-Language` (negocia com `locales` suportados)
3. `defaultLocale = 'pt-BR'`

Acessos à raiz (`/`) redirecionam para `/<locale>/` conforme essa ordem.

### Switcher

- Componente `<LocaleSwitcher />` em `components/shared/locale-switcher.tsx` (client component)
- Dois botões compactos no footer: `PT` e `EN`
- Ao clicar:
  1. `document.cookie = 'NEXT_LOCALE=<locale>;path=/;max-age=31536000;samesite=lax'`
  2. `router.replace(pathname, { locale })` — navega mantendo a rota atual
- Estado ativo visualmente destacado (bg cyan); desabilitado enquanto pending via `useTransition`

### Estrutura de keys

Hierarquia por domínio. Keys em **inglês, snake_case ou camelCase** conforme convenção do namespace:

```
common.{actions,footer,localeSwitcher}.*
metadata.{siteName,siteDescription,termsTitle,...}
home.{sprintBadge,heroTagline,cards.{phases,scoring,xp}.{title,description,body},...}
legal.{lastUpdated,brandLabel,terms.{title,lastUpdated,sections[]},privacy.{...}}
game.*         (futuro, TASK-013/014)
daily.*        (futuro, TASK-016)
chat.*         (futuro, TASK-013)
bot.*          (futuro, TASK-010/011) — mensagens do Sound Bot
profile.*      (futuro, TASK-008)
room.*         (futuro, TASK-012)
xp.*           (futuro, TASK-031)
admin.docs.*   (futuro, TASK-029)
```

### ICU para plurais, datas, números

Sempre que há contagem variável, usar plural ICU:

```json
// pt-BR.json
"game": {
  "players_count": "{count, plural, one {# jogador} other {# jogadores}}"
}
// en.json
"game": {
  "players_count": "{count, plural, one {# player} other {# players}}"
}
```

Uso: `t('game.players_count', { count: 5 })` → "5 jogadores" / "5 players".

### Como adicionar uma nova string

1. Identificar namespace adequado
2. Adicionar em **ambos** `messages/pt-BR.json` **e** `messages/en.json` com a mesma estrutura
3. Referenciar via `useTranslations('<namespace>')` (client) ou `getTranslations({ locale, namespace })` (async server)
4. Em server components não-async, usar `useTranslations()` diretamente — funciona graças ao React context da RSC (next-intl docs)

### Validação de cobertura

Em CI, rodar `pnpm docs:check` que (quando TASK-029 acrescentar) inclui: comparar chaves de `pt-BR.json` vs `en.json`, falhar se houver discrepância. Por enquanto é validação manual.

### Metadata traduzida

Todo `Metadata` gerado por página usa `generateMetadata({ params })`:

```tsx
export async function generateMetadata({ params }): Promise<Metadata> {
  const { locale } = await params;
  const t = await getTranslations({ locale, namespace: 'metadata' });
  return { title: t('termsTitle'), description: t('termsDescription') };
}
```

### `<html lang>` dinâmico

`app/[locale]/layout.tsx` renderiza `<html lang={locale}>`. Não há root `app/layout.tsx` — toda rota passa pelo locale layout.

## Requisitos Não-Funcionais

- Build gera SSG para cada rota × cada locale (atualmente 3 rotas × 2 locales = 6 páginas estáticas)
- Middleware `51.6 kB` gzipped — dentro do orçamento de edge do Vercel
- Zero flash de conteúdo errado: pages são SSR por locale, conteúdo já chega renderizado no idioma correto
- CLS zero: fonts via `next/font` com `display: swap`

## Regras para Specs de Features (04/05/06 e futuras)

Specs que referenciam UI text **devem** mostrar exemplos como keys, não strings cruas:

❌ Errado:
```markdown
Bot message: `🎵 Rodada 3 de 10 — Categoria: Rock`
```

✅ Certo:
```markdown
Bot message: key `bot.round_announcement`
  pt-BR: `🎵 Rodada {round} de {total} — Categoria: {category}`
  en:    `🎵 Round {round} of {total} — Category: {category}`
```

Isso mantém as specs fontes-de-verdade sincronizadas com o código.

## Edge Cases

- **Acessar `/` (sem prefixo):** middleware redireciona com 307 para `/<locale>/` baseado em cookie > Accept-Language > pt-BR
- **Acessar locale inválido (ex: `/fr/`):** middleware roteia como não-locale → 404
- **Cookie apontando para locale removido futuramente:** middleware ignora cookie inválido e cai no fallback
- **Client troca de locale com JS desabilitado:** degradação ok — apenas o switcher não funciona; usuário pode editar URL manualmente
- **next/font + locale:** todos os locales compartilham as mesmas fontes (latin subset cobre pt-BR e en); se adicionar línguas com escrita diferente no futuro, subset por locale

## Decisões de Design

- **Prefixo por locale, não cookie puro:** URLs compartilháveis preservam a língua; bom para SEO e OG images; next-intl recomenda.
- **next-intl, não lingui nem custom:** padrão do ecossistema Next.js App Router em 2026, RSC-first, menor boilerplate.
- **Default pt-BR:** público primário brasileiro; en é fallback secundário.
- **Sem internacionalização de moeda/data no MVP:** o jogo não mostra moeda; datas já vêm formatadas de forma neutra (`2026-04-17`). Quando aparecerem (ex: DailyCountdown), usar `t.dateTime(date, format)` do next-intl.
- **Keys em inglês:** um dev que não lê português consegue navegar o código e as keys; traduções ficam no JSON.
