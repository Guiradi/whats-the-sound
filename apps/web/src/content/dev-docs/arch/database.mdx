## RLS (resumo)

**Todas as tabelas têm RLS ativado.**

- `users`: read all, update own, insert via trigger (SECURITY DEFINER) no OAuth
- `midi_catalog`: public read de `is_active=true`, admin write
- `game_sessions` / `game_players` / `round_scores`: read public (rankings), write via service role
- `daily_results`: read own + public leaderboard, write via service role
- `xp_events`: read own, write via service role
- Storage `midis` bucket: public read, admin write

## Triggers importantes

- `handle_new_user` (on `auth.users` insert): cria row em `public.users` com email + nickname sanitizado do provider metadata (strip non-alphanumeric, retry com sufixo `_N` se já existir) ou fallback `player_<id8>`
- `sync_user_level` (on `users.xp` update): `level = floor(sqrt(xp/100)) + 1`
- `update_user_stats_after_game` (on `round_scores` insert): incrementa counters do jogador
- `update_daily_streak` (on `daily_results` insert): detecta streak contínua (ontem + hoje)

## Colunas sensíveis

- **`midi_catalog.accepted_titles` / `accepted_artists`**: NUNCA enviar ao client antes de `round:reveal`. Verificação server-side só.
- **`users.email`**: PII — não expor em APIs públicas. Visível apenas para o próprio usuário e admins.
- **`users.role`**: enum `(player, admin)`. Admin se concede via SQL manual (ver [Runbook](/admin/docs/runbooks/supabase-admin)).
